# Дашборд воронки продаж

Встроенное приложение Битрикс24 для пилота партнёрской компетенции VibeCode. Оно показывает сводку по стадиям выбранной воронки, ключевые показатели и 15 последних сделок за период создания.

Публичный репозиторий с исходным кодом: <https://github.com/krotikovmgk/bitrix24-vibecode-sales-dashboard>

## Возможности

- фильтры по воронке, периоду создания и ответственному («все доступные» / «только мои»);
- количество и сумма сделок по каждому `stageId`;
- сумма открытых сделок, число выигранных и средний чек выигранных;
- последние сделки: название, сумма, стадия, ответственный и дата;
- прозрачная авторизация текущего пользователя через VibeCode Gateway;
- корректное раздельное отображение сумм, если в CRM есть несколько валют;
- адаптивный интерфейс для встраивания в Битрикс24.

## Архитектура и безопасность

Браузер обращается только к BFF этого приложения. Gateway передаёт серверу короткоживущую пользовательскую сессию в заголовке `X-Vibe-Authorization`; сервер пересылает её Vibe API как `Authorization: Bearer`. Ключ приложения читается только из `VIBE_APP_KEY` и не попадает в HTML, JavaScript, логи или репозиторий.

Для повторных запросов фильтра BFF выдаёт браузеру случайный непрозрачный идентификатор, действующий не более четырёх часов бездействия. OAuth-токен хранится только в памяти серверного процесса и никогда не передаётся клиентскому JavaScript или browser storage. После перезапуска контейнера обычное обновление встроенного приложения создаёт новую сессию.

Запрошенные скоупы:

- `crm` — чтение сделок, воронок и стадий;
- `user` — имя ответственного и контекст сотрудника;
- `placement` — публикация приложения и пункт в интерфейсе Битрикс24.

## Локальный запуск

Требуется Node.js 20+.

```bash
npm test
VIBE_APP_KEY='vibe_app_…' npm start
```

Проверка доступности:

```bash
curl http://127.0.0.1:3000/health
```

Без Gateway-заголовка API дашборда ожидаемо вернёт `SESSION_MISSING`: рабочий пользовательский контекст появляется при открытии опубликованного приложения из Битрикс24.

## Воспроизводимый деплой VibeCode

1. Создать OAuth-приложение через `POST /v1/apps` со скоупами `crm`, `user`, `placement`; сохранить одноразовый `rawKey` в менеджере секретов.
2. Собрать gzip-tar архив этой директории без `.env`, архивов и служебных файлов.
3. Создать Galaxy-приложение одним `POST /v1/infra/servers` с `runtime: node20`, `start: node server.js`, `port: 3000`, `env.VIBE_APP_KEY` из менеджера секретов и `source.content` с base64-архивом.
4. Задать полученный `appUrl` вызовом `PATCH /v1/apps/:id`.
5. Один раз авторизовать OAuth-приложение на целевом портале.
6. Опубликовать его вызовом `POST /v1/apps/:id/publish` с поддерживаемым размещением из `GET /v1/placements/available`.
7. Открыть приложение в Битрикс24 под администратором и обычным сотрудником.

Развёрнутый экземпляр: <https://app-4c81d058ac8d.vibecode.bitrix24.tech>  
OAuth App ID: `e4a130fd-f694-4f51-95a5-d909b85956fb`  
Портал: `virtbridge.bitrix24.ru`, пункт меню «VirtBridge — Дашборд воронки продаж».

Для получения пользовательского CRM-контекста приложение следует открывать через пункт меню авторизованного портала. Политика `PORTAL` намеренно не даёт анонимному внешнему посетителю доступ к данным CRM по прямой ссылке.

## Ограничения

- за один отчёт обрабатывается до 5 000 сделок; UI явно сообщает об усечении;
- суммы разных валют не конвертируются, а показываются раздельно;
- период применяется к дате создания сделки;
- видимость данных определяется CRM-правами текущего пользователя Битрикс24.
